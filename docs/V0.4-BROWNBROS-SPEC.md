# Consite v0.4 — Brown Bros Operations Spec

**IP:** Consite is Jungle Labs Inc. intellectual property, delivered to Brown Bros
Framing and Drywall as a service under the retainer (Section 4). Nothing in this
build is client-owned except their data.

**Source:** How Brown Bros actually runs a site (from the owner, July 2026).

---

## The crew model (per project)

| Role | Who | What they do in Consite |
|---|---|---|
| Crew (×3) | Workers | Clock in/out, complete assigned pin-tasks, upload work photos + voice notes, report problems |
| Foreman | Runs the crew | Approves crew hours, owns the daily FLHA + site forms, pins tasks on drawings, assigns work |
| Lead foreman | Above foremen | Everything a foreman does, across crews; manages foremen |
| Admin | Office | Users, projects, payroll export, compliance, records |
| Manager | Above admin | Sees everything on their projects; read-heavy dashboard |

Mapping: `ProjectMember.role` = `worker` / `foreman` / `lead-foreman` (per project).
Admin + manager are top-level accounts (`users.role`), not per-project.

## Feature 1 — Hours approval  ✅ SHIPPED v0.4.0

- Clock-out sets shift `status: pending`
- Foreman's **Crew Hours** screen lists pending crew shifts → one-tap approve
- No self-approval (enforced in security rules + UI)
- Auto-closed shifts (missed clock-out) arrive flagged ⚠ for review
- Payroll CSV labels every row `approved` / `PENDING FOREMAN APPROVAL` / `NO CLOCK-OUT — REVIEW`
- Admin promotes workers via role chip in Users → tap to cycle Worker → Foreman → Lead Foreman

## Feature 2 — Work proof: photos + voice (next)

Worker flow: take photo(s) of completed work → hold-to-talk voice note ("finished
boarding the north wall, second floor") → AI structures it into a work entry
(what, where, quantity) → attached to project timeline under worker + date + GPS.

- Data: `projects/{pid}/workLog/{id}` — photos (Storage), transcript, AI summary, uid, ts
- Surfaces: project timeline for foreman/admin/manager; daily log generator consumes it

## Feature 3 — Site drawings + pin tasks (the big one)

- Foreman uploads drawings (PDF/image) per project (`plans` collection exists since v0.1)
- Foreman taps a point on the drawing → creates a **task pin**: photo of what's expected,
  voice/text instruction, assigned worker, due date
- Worker sees "My Tasks" → opens pin → does work → uploads completion photo + voice note
- Foreman reviews → accepts or bounces back with note
- If a worker finds a problem instead: same pin flow but flagged as **issue** (merges with deficiency system)
- Data: `projects/{pid}/plans/{planId}/pins/{pinId}` — x/y (0–1 normalized), type task|issue,
  status open|in-progress|done|accepted, assigneeUid, photos[], voiceNote, history[]

## Feature 4 — FLHA is the foreman's duty + auto-records

- FLHA reassigned: **one per site per day, owned by the foreman** (not every worker)
- Crew members countersign on the foreman's phone (signature capture) after toolbox talk
- On submit: PDF rendered server-side → **auto-emailed to the office** with subject
  `FLHA · [Project] · [Date] · [Foreman]` (decision: email now, Drive sync phase 2)
- Same record pipeline for the other forms: environmental checklist, **quality check (QC)**,
  orientation, toolbox talk — the form renderer already supports 7 field types;
  each form type = a `forms/{formId}` schema + per-site required-paperwork rules
- Brown Bros' actual FLHA PDF (received July 8) becomes the canonical `flha-brownbros-v1` schema

## Feature 5 — Manager/admin visibility

- Manager sees per-project: who's on site, hours (approved vs pending), work-log timeline,
  pin-task board, FLHA/QC completion, open issues
- Admin additionally: payroll export, user management, records archive
